# SX6036 スイッチ内蔵 SM 不具合調査レポート

- **実施日時**: 2026年2月25日 21:49〜22:45
- **関連レポート**: [report/2026-02-25_203745_ib_switch_benchmark.md](2026-02-25_203745_ib_switch_benchmark.md)

## 前提・目的

前セッションで SX6036 の内蔵 Subnet Manager を `ib sm` で有効化したが、SM プロセスが正常動作せず server 5 のポートが Initialize のまま Active に遷移しなかった。opensm (server 4) へのフォールバックで解決したが、根本原因が不明なため再調査する。

### 調査目的

1. SM プロセスが起動しない根本原因を特定する
2. スイッチ SM で運用可能かを判定する
3. 不可の場合は opensm 運用の正当性を確認する

## 環境情報

| 項目 | 値 |
|------|-----|
| スイッチ | Mellanox SX6036 (MSX6036F-1SFS) |
| MLNX-OS | 3.6.8008 (2018-07-03) |
| 内蔵 SM | OpenSM 4.8.1.MLNX20171015.0395770 |
| server 4 opensm | 3.3.23 (systemd, enabled) |
| server 4 | 10.10.10.204 (IPoIB 192.168.100.1) |
| server 5 | 10.10.10.205 (IPoIB 192.168.100.2) |
| ポート接続 | IB1/23→server 4, IB1/21→server 5 |
| 接続方法 | シリアルコンソール (USB, /dev/ttyUSB0, 9600/8N1) |

### Phase 2: opensm 停止 → スイッチ SM 有効化

#### 実行手順

1. `systemctl stop opensm` (server 4)
2. `configure: ib sm` (スイッチ SM 有効化)
3. 60 秒待機
4. 診断情報収集

#### 結果

| 診断項目 | 結果 |
|---------|------|
| show ib sm | **enable** |
| show ib sm routing-info | **Connection to SM unavailable** |
| show ib sm log | **Error reading ib sm log file** |
| server 4 ibstat | Active (stale LID 残留) |
| server 5 ibstat | Active (stale LID 残留) |

**前回と完全に再現。** SM 設定は enable だが、プロセスが動作していない。

#### 重要な発見: smnode の2層構造

`show ib smnode switch-d2b2e2 ?` で詳細な SM ノード診断コマンドを発見:

| コマンド | 結果 | 意味 |
|---------|------|------|
| sm-state | **enabled** | SM 設定は有効 |
| **sm-running** | **not active** | **SM プロセスが実行されていない** |
| sm-licensed | active | ライセンスはある |
| active | active | HA ノードはアクティブ |
| ha-role | master | マスターノード |
| ha-state | online | オンライン |
| ip | 10.125.13.104 | mgmt0 IP |

**根本的な問題**: SM は設定上 enabled だが、プロセスが起動しない。

#### smnode enable の追加テスト

running-config で `ib smnode switch-d2b2e2 disable` だったため、`enable` に変更:
```
configure: ib smnode switch-d2b2e2 enable
```
→ 30 秒後も `sm-running: not active` のまま。

`ib sm` のトグル (no ib sm → ib sm) も効果なし。

### Phase 3: 仮説テスト

#### H2: SM priority を 15 に上げる

```
configure: ib sm sm-priority 15
```
→ 30 秒後も `sm-running: not active`。**否定。**

#### H3: sweep interval を 5 に変更

```
configure: ib sm sweep-interval 5
```
→ 20 秒後も `sm-running: not active`。**否定。**

#### システムログ分析 (`show log`) — 根本原因特定

```
Aug 14 00:11:22 switch-d2b2e2 kernel: end_request: I/O error, dev mtdblock6, sector 464
Aug 14 00:11:22 switch-d2b2e2 kernel: Buffer I/O error on device mtdblock6, logical block 58
Aug 14 00:11:22 switch-d2b2e2 kernel: end_request: I/O error, dev mtdblock7, sector 1048512
Aug 14 00:11:22 switch-d2b2e2 kernel: Buffer I/O error on device mtdblock7, logical block 131064
...
Aug 14 00:16:26 switch-d2b2e2 cli[8023]: chc_smm_update: Unable to get SMM running status
...
Aug 14 00:20:25 switch-d2b2e2 cli[8131]: chc_smm_update: Unable to get SMM running status
```

**発見した問題:**

1. **mtdblock6/mtdblock7 の I/O エラー**: スイッチ内蔵フラッシュストレージに物理的なエラーが発生している
2. **SMM (Subnet Manager Manager) ステータス取得失敗**: `chc_smm_update: Unable to get SMM running status` が繰り返し発生
3. **mgmtd タイムアウト**: `Async: timed out getting external response for type query_request ... from pm-4381` — SM プロセス (pm-4381) への問い合わせが常にタイムアウト

SM 有効化後、シリアルコンソールが以下のメッセージで完全にフラッディングされた:
- `mgmtd` による SM プロセスへのポーリングタイムアウト (毎分)
- `chc_smm_update: Unable to get SMM running status` (毎分)
- フラッシュ I/O エラー (リアルタイムで発生: `Feb 22 13:39:50 ... end_request: I/O error, dev mtdblock7`)

### Phase 4: 復旧

通常の `sx6036-console.py` ではコンソールのフラッディングにより認証不可。
ノイズ耐性のある専用復旧スクリプト (`force-restore.py`) を作成して実行:

1. シリアルバッファをドレイン
2. Ctrl+C でブレーク
3. コマンドを直接送信 (プロンプト検出に依存しない)
4. `no ib sm`, `ib sm sm-priority 0`, `ib sm sweep-interval 10`, `ib smnode switch-d2b2e2 disable`
5. `write memory` で永続化

復旧後の確認:

| 項目 | 結果 |
|------|------|
| show ib sm | disable |
| running-config | smnode disable, priority 0 (デフォルト復旧) |
| opensm restart | 成功 (systemctl start opensm) |
| server 4 ibstat | Active, lid=4 |
| server 5 ibstat | Active, lid=1 |
| IPoIB ping (192.168.100.2) | 3/3 成功 (0% loss) |

## 結論

### 根本原因

SX6036 スイッチ内蔵 SM が動作しない原因は **フラッシュストレージ (mtdblock6/mtdblock7) の I/O エラー**。

SM プロセスは起動時にフラッシュ上のログファイルやワーキングデータにアクセスするが、I/O エラーにより初期化に失敗し、`sm-running: not active` のまま停止する。設定レイヤ (`ib sm`, `ib smnode`) では enabled と報告されるが、実際のプロセスは決して起動しない。

### 仮説の検証結果

| 仮説 | 結果 | 詳細 |
|------|------|------|
| H1: SM クラッシュ | **部分的に正しい** | プロセスは起動すらしない (クラッシュ以前の問題) |
| H2: priority 0 が原因 | **否定** | priority 15 でも変化なし |
| H3: sweep interval が不適切 | **否定** | interval 5 でも変化なし |
| H4: smnode 設定の干渉 | **否定** | smnode enable しても変化なし |
| H5: ポート固有の問題 | **判定不能** | SM 不在で SMP クエリ不可 |
| **H6: FW/HW 問題** | **確認** | **mtdblock I/O エラーが根本原因** |

### 運用判断

- **スイッチ SM は使用不可** — ファームウェア更新 (3.6.8008 → 最新) またはフラッシュ修復なしでは解決しない
- **opensm (server 4) を正式 SM として運用する** — 現在の構成 (`systemctl enable opensm`) を維持
- スイッチの `ib sm` 設定は `disable` のまま `write memory` 済み

### 補足: SM 有効化時のリスク

SM を有効化すると以下の副作用が発生する:
1. SM プロセスの起動失敗がコンソールをフラッディングする (mgmtd タイムアウトメッセージ)
2. シリアルコンソールの通常操作が困難になる
3. 復旧には専用のノイズ耐性スクリプトが必要

## 再現方法

1. server 4 の opensm を停止: `systemctl stop opensm`
2. スイッチで SM 有効化: `configure: ib sm` + `ib smnode switch-d2b2e2 enable`
3. 60 秒待機
4. `show ib smnode switch-d2b2e2 sm-running` → `not active` を確認
5. `show ib sm log` → `Error reading ib sm log file` を確認
6. 復旧: `configure: no ib sm` + `ib smnode switch-d2b2e2 disable` → `systemctl start opensm`
