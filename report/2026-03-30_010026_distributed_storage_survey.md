- **実施日時**: 2026年3月30日 01:00 (JST)

## 添付ファイル

## 前提・目的

### 背景

LINSTOR/DRBDで2リージョン（各3ノード）のマルチリージョンVM運用基盤を構築・評価してきた。以下の実験・調査で、LINSTORの特性が明らかになった:

- [LINSTOR RAID5 resilience 実験](2026-03-22_120534_linstor_raid5_resilience_experiment.md) — DRBD層の縮退運転は可能だが、LINSTOR自体はノード内ディスク冗長をサポートしない
- [LINSTOR Software RAID1 実験](2026-03-22_220051_linstor_software_raid1_experiment.md) — LINSTORのRAID1機能はノード間レプリケーションであり、ノード内冗長ではない
- [LINSTOR Per-disk Pool 調査](2026-03-28_000658_linstor_per_disk_pool_investigation.md) — 1ディスク1プール構成でもノード内冗長にはならない
- [LINSTOR LVM-RAID1 ベンチマーク](2026-03-29_090042_linstor_lvm_raid1_benchmark.md) — LVM-mirrorで冗長化は可能だが管理層が増える

### 目的

LINSTOR/DRBD以外の分散ストレージで、以下の2要件を**組み込みで**満たすものを調査する:

1. **縮退運転**: ノード障害時にサービスを継続できること（LINSTORと同等）
2. **ノード内ディスク冗長**: 1ノード内の単一ディスク障害に対して、別途RAID/LVMを組まずに保護できること

### 前提条件

- Proxmox VE 9.1.6 クラスタ（6ノード、2リージョン x 3ノード）
- Region A: 4+5+6号機 (Supermicro X11DPU)、Region B: 7+8+9号機 (Dell R320)
- 用途: VMブロックストレージ（ライブマイグレーション必須）
- オープンソースまたはラボ利用可能なライセンス

## 調査結果

### 要件を満たす候補

9つの分散ストレージを調査した結果、縮退運転とノード内ディスク冗長の**両方を組み込みでサポートする**のは **Ceph** と **Vitastor** の2つのみだった。加えて、現行のLINSTOR + LVM-mirrorも実用的な選択肢として残る。

#### 1. Ceph — 安全な選択肢

**ノード内ディスク冗長の仕組み**: 1物理ディスク = 1 OSD（JBOD, RAID不要）。CRUSHマップの`failure_domain=host`で各レプリカを異なるホストに配置する。ディスク(OSD)障害時はクラスタ全体で自動再配置が行われ、手動介入なしに冗長性が回復する。

| 項目 | 評価 |
|------|------|
| 縮退運転 | `min_size=2` で1ノード障害時も継続稼働 |
| ノード内ディスク冗長 | OSD単位の自動再配置で実現 |
| 性能 (4K random) | read ~8K IOPS, write ~1.5K IOPS (LINSTORの1/2〜1/5) |
| CPU使用量 (飽和時) | ~40コア/ノード |
| PVE統合 | **組み込み** (pveceph CLI, GUI管理, HA, スナップショット) |
| 3ノードEC | 非実用的 (host domainではEC 2+1で冗長性確保不可) |
| 容量効率 (3xレプリカ) | 33% |
| マルチリージョン | Stretch cluster (構成が複雑) |
| コミュニティ | 大 (Red Hat/IBM支援) |
| ライセンス | LGPL |

**3ノード運用時の注意点**:
- OSD使用率は45%以下に抑える（回復ヘッドルーム確保）
- ノード全損後は自動回復不可（4番目のホストがないため3番目のレプリカを配置できない）
- 5ノード以上が推奨される構成

#### 2. Vitastor — 高性能な新興候補

**ノード内ディスク冗長の仕組み**: Cephと同じOSD/diskモデル。配置ツリーで`failure_domain`を設定。`failure_domain=host`でノード障害保護とディスク障害保護を同時に実現。CRUSH配置の均一性がCephより優秀（96-99% vs 80-90%）。

| 項目 | 評価 |
|------|------|
| 縮退運転 | `pg_minsize` で縮退運転を設定可能 |
| ノード内ディスク冗長 | OSD単位の自動再配置で実現 |
| 性能 (4K random) | latency 0.14ms, read 895K IOPS (NVMe+RDMA時) |
| CPU使用量 (飽和時) | ~4コア/ノード (**Cephの1/10**) |
| PVE統合 | 3rdパーティプラグイン (pve-storage-vitastor) |
| 3ノードEC | **可能** (`failure_domain=osd` でOSD単位分散) |
| 容量効率 (EC 2+1) | **67%** (Ceph 3xレプリカの2倍) |
| マルチリージョン | 未対応 |
| コミュニティ | 小 (GitHubスター ~700) |
| ライセンス | VNPL 1.1 (copyleft, ラボ利用可) |

**最大のリスク**: 開発者1名 (Vitaliy Filippov) — バスファクター1。プロジェクト開始は2020年頃。

**3ノードEC運用の特長**:
- `failure_domain=osd` + EC 2+1 で、3ノード各4ディスク(=12 OSD)に分散
- 任意の1 OSD障害に耐える + 容量効率67%
- `failure_domain=host` にすればノード障害にも耐えるが、3ノードでは実質3xレプリカと同等

**意図的な縮退運転（省電力目的）**:

オフピーク時に1〜2ノードを意図的にシャットダウンし、消費電力を削減するシナリオについて調査した。

**3ノード → 2ノード運転: 実用的**

`noout` フラグを使うことで、不要なリバランスなしにノードを停止・復帰できる:

```sh
# 停止前: 対象ノードのOSDにnooutを設定（リバランス抑制）
vitastor-cli modify-osd --noout true <OSD_1> <OSD_2> ...
# ノードをシャットダウン
# → プールはpg_minsize=2で縮退運転を継続（レプリカ/EC問わず）

# 復帰時: ノード起動後にnooutを解除
vitastor-cli modify-osd --noout false <OSD_1> <OSD_2> ...
# → 停止中に書き込まれたデータのみ同期（最小限のI/O）
```

- レプリケーション (pg_size=3, pg_minsize=2): 2コピーで運転。冗長性は低下するが可用
- EC 2+1 (pg_minsize=2): 2チャンク中2チャンク利用で運転。1チャンク欠損でも読み取り可能
- `noout` を設定しない場合、`osd_out_time`（デフォルト600秒）経過後に自動リバランスが開始され、復帰時に再度リバランスが発生する（2回分のI/Oが無駄になる）

**3ノード → 1ノード運転: 技術的には可能だが実用的でない**

| 制約 | 詳細 |
|------|------|
| **etcdクォーラム喪失** | 3ノードetcdクラスタでは2ノード停止でクォーラムを失い、Vitastor全体が停止する。外部etcdまたはwitnessノードが必要 |
| **ECプール不可** | EC 2+1は最低2チャンク必要。1ノード(=1/3のOSD)では復元不可能。ECプールは完全に停止する |
| **レプリカプールのみ** | `pg_minsize=1` に設定すればレプリケーションプールは1コピーで稼働可能。ただし冗長性ゼロ — ディスク障害 = データ損失 |
| **プール設定の動的変更** | `pg_size`, `pg_minsize`, `failure_domain` はライブで変更可能（`vitastor-cli modify-pool`）。ただし `scheme`（replicated/EC）は変更不可 |

1ノード運転を実現するには:
1. ECではなくレプリケーションプールを使用する（ECは1ノードで動作不可）
2. etcdを外部ノード（軽量VM等）に配置するか、witnessノードを用意する
3. `pg_minsize=1` に変更する（冗長性ゼロを受容する）

**省電力縮退の実用性まとめ**:

| シナリオ | 可否 | 条件・制約 |
|---------|------|-----------|
| 3→2ノード (レプリカ) | **実用的** | `noout` + `pg_minsize=2`。冗長性低下するが安全に運用可能 |
| 3→2ノード (EC 2+1) | **実用的** | 同上。EC 2+1は1ノード欠損に耐える設計 |
| 3→1ノード (レプリカ) | 条件付き | 外部etcd必須 + `pg_minsize=1` + 冗長性ゼロ（非推奨） |
| 3→1ノード (EC) | **不可** | 最低2チャンク必要、1ノードでは復元不可能 |

#### 3. LINSTOR + LVM-mirror — 現行改善

| 項目 | 評価 |
|------|------|
| 縮退運転 | DRBD 2+1トポロジーで実現済み |
| ノード内ディスク冗長 | LVM-mirror (RAID1相当) で実現 |
| 性能 | 最高 (LVM-mirrorのオーバーヘッドは小) |
| PVE統合 | ネイティブプラグイン |
| マルチリージョン | **対応済み** (唯一) |
| 追加管理コスト | LVM-mirrorの障害対応は手動 |
| 容量効率 | 25% (ノード内RAID1 + ノード間2xレプリカ) |

| システム | 不適格理由 |

| GlusterFS | PVE 9 (2025年7月) でサポート削除、QEMUドライバも非推奨化 |
| LizardFS | 2021年頃から開発停滞、事実上のプロジェクト終了 |
| SeaweedFS | オブジェクト/ファイルストレージのみ、VMブロックストレージ非対応 |
| Longhorn | Kubernetes専用、ベアメタルPVE非対応 |
| OpenEBS | Kubernetes専用、ベアメタルPVE非対応 |
| DAOS | HPCオブジェクトストア、VMブロックストレージ非対応 |
| MinIO | オブジェクトストレージのみ |

## 結論

### 選択肢の整理

3つの現実的なパスがある:

1. **Ceph**: 安定性とPVE統合を最優先する場合。性能はLINSTORの1/2〜1/5に低下するが、組み込みサポートによる運用の容易さと大規模コミュニティによる長期的な安心感がある。

2. **Vitastor**: 性能とノード内冗長の両立を最優先する場合。LINSTOR並みの性能を維持しつつ、Ceph同様のOSD/diskモデルでノード内冗長を実現。3ノードECによる容量効率67%も魅力。省電力目的の3→2ノード縮退は`noout`フラグで実用的に運用可能。3→1ノードはECプール不可・外部etcd必須・冗長性ゼロのため非推奨。バスファクター1のリスクが大きい。

3. **LINSTOR + LVM-mirror**: 移行コスト最小で現行マルチリージョン構成を維持する場合。既にベンチマーク済みで実用性は確認されている。
