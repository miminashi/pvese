# SX6036 ファームウェア更新・フラッシュ修復レポート

- **実施日時**: 2026年2月26日 00:00 〜 01:11
- **関連 Issue**: #22 SX6036 ファームウェア更新・フラッシュ修復

## 前提・目的

### 背景

SX6036 InfiniBand スイッチの内蔵 Subnet Manager (SM) が動作しない問題を調査した結果（[前回レポート](2026-02-25_224551_sx6036_sm_investigation.md)）、根本原因がフラッシュストレージ (mtdblock6/mtdblock7) の I/O エラーであると特定した。

### 目的

1. ファームウェアを 3.6.8008 → 3.6.8012（最終版）に更新する
2. `image install` の Step 3 (Create Filesystems) によるフラッシュの再フォーマットで論理的破損を修復する
3. 修復後に内蔵 SM の動作を確認する

### 前提条件

- スイッチへの管理アクセスはシリアルコンソールのみ（server 4 経由の USB シリアル）
- mgmt0 (10.125.13.104/24) はラボネットワーク (10.10.10.0/24) から到達不可
- server 4 の opensm でIBファブリックを運用中

## 実施手順と結果

### Phase 1: mgmt0 ネットワーク接続の確立

**目的**: FW イメージ転送のため mgmt0 への IP 到達性を確保する。

1. mgmt0 のルーティングテーブルを確認:
   - IP: 10.125.13.104/24, デフォルトゲートウェイ: 10.125.13.1
   - ラボネットワーク (10.10.10.0/24) とは別サブネット
   - `ping -c 3 10.10.10.204` → Destination Host Unreachable

2. mgmt0 IP をラボサブネットに変更:
   ```
   configure terminal:
     no interface mgmt0 dhcp
     interface mgmt0 ip address 10.10.10.100 /24
     no ip route 0.0.0.0/0 10.125.13.1
   ```

3. 検証:
   - server 4 → `ping 10.10.10.100` → 成功 (0.3ms)
   - mgmt0 がラボネットワークと同じ L2 にあることを確認

4. `write memory` で永続化

**結果**: mgmt0 を 10.10.10.100/24 に変更し、ラボネットワークから到達可能にした。

### Phase 2: ファームウェアイメージの取得

1. `wget` で content.mellanox.com から FW イメージをダウンロード (358MB)
2. `scp` で server 4 の /tmp/ に転送

### Phase 3: 更新前の準備

1. `show running-config` で設定をバックアップ
2. `show images` で現在のイメージ状態を記録:
   - Partition 1: 3.6.8008 (ブート中)
   - Partition 2: 3.6.8008
3. `show log` でフラッシュエラー状態を記録（mtdblock6/mtdblock7 I/O エラー多数）
4. server 4 の opensm を停止: `systemctl stop opensm`

### Phase 4: ファームウェア更新

#### 4.1: FW イメージのスイッチへの転送

- server 4 で一時 HTTP サーバ (`python3 -m http.server 8080`) を起動
- スイッチ側: `image fetch http://10.10.10.204:8080/image-PPC_M460EX-3.6.8012.img`
- ダウンロード完了まで約 5 分

#### 4.2: Partition 2 にインストール

```
image install image-PPC_M460EX-3.6.8012.img
```

- デフォルトで次のブートパーティション（Partition 2）にインストール
- `location` キーワードで指定可能（`partition` ではない）
- 4 ステージで処理:
  1. Step 1 of 4: Verify Image
  2. Step 2 of 4: Uncompress Image
  3. Step 3 of 4: Create Filesystems ← **フラッシュ再フォーマット**
  4. Step 4 of 4: Extract Image
- 所要時間: 約 15 分 (910 秒)

#### 4.3: ブートパーティション切替・リブート

```
image boot next        → Next boot partition: 2
write memory
reload                 → y
```

- リブート所要時間: 約 8.5 分 (510 秒)

#### 注意点

- リブート後にシリアルコンソールが無応答になった（全ボーレートで無応答）
- 原因: DTR/RTS シグナルのリセットが必要だった
- `ser.dtr = False; ser.dtr = True` でコンソール復帰
- 最初にログインした際の `CLI > ` プロンプトは、前のセッションの残り（cli_hide_cmds モジュール初期化失敗による限定シェル）
- `exit` → 再ログインで正常なプロンプト (`switch-d2b2e2 [standalone: master] >`) に復帰

### Phase 5: 更新後の検証

#### 5.1: FW バージョン確認

```

Build date:        2019-02-22 07:53:42
Version summary:   PPC_M460EX 3.6.8012 2019-02-22 07:53:42 ppc
```

#### 5.2: フラッシュ健全性チェック — **修復成功**

`show log` の検査結果:
- **mtdblock I/O エラー: なし** (更新前は多数発生)
- **カーネルエラー: なし**

FW 更新の Step 3 (Create Filesystems) がフラッシュのファイルシステムを再フォーマットし、論理的破損が修復された。

#### 5.3: SM 動作テスト — **動作確認**

```

  ib smnode switch-d2b2e2 enable
```

60 秒待機後:

`write memory` で永続化。

#### 5.4: IB ファブリック検証

#### 5.5: opensm 無効化

## 知見・注意事項

### MLNX-OS image install の構文

- `image install <file>` — 次のブートパーティションに自動インストール
- `image install <file> location <N>` — 指定パーティションにインストール
- `partition` キーワードは MLNX-OS 3.6 では使用不可（エラーになる）

### シリアルコンソールの注意

- FW 更新後のリブートでシリアルコンソールが無応答になることがある
- DTR/RTS シグナルのトグル (`ser.dtr = False → True`) で復帰可能
- リブート直後の最初のログインセッションは CLI モジュールの初期化に失敗し、限定シェル (`CLI >`) になることがある → `exit` して再ログインで解決

### image fetch の進捗バー

- 進捗バーに `#` 記号が含まれるため、シリアルコンソール自動化スクリプトのプロンプト検出 (`# `) と衝突する
- 対策: 完了後のフルプロンプトパターン (`switch-d2b2e2 [standalone: master] #`) で検出するか、進捗率の数値を監視する

### SSH 接続

- mgmt0 への SSH は古い鍵交換アルゴリズム (diffie-hellman-group14-sha1) が必要
- SSH サーバ: mpSSH_0.2.1
- パスワード認証: リブート直後は一時的に失敗することがある（システム初期化中）

## 正しい手順の発見経緯

自動化スクリプトによるシリアルコンソール操作では、MLNX-OS 特有の挙動により複数の問題に遭遇した。以下にその経緯を記録する。

### 1. mgmt0 への到達性確保

当初 mgmt0 は 10.125.13.104/24 で、ラボネットワーク (10.10.10.0/24) から到達不可だった。ルーティングテーブル (`show ip route`) とスイッチからの ping テストで到達不可を確認した後、mgmt0 が同じ L2 にある可能性を考えて IP をラボサブネットに変更した。`write memory` せずに変更し、接続を確認してから永続化するアプローチで、失敗時のリスクを最小化した。

### 2. `image install` の構文問題

計画では `image install <file> partition 2` の構文を想定していたが、MLNX-OS 3.6 では `partition` キーワードは存在しなかった。`image install <file> ?` でヘルプを表示したところ、正しいキーワードは `location` であることが判明した。また、引数なしの `image install <file>` がデフォルトで「次回ブートパーティション」にインストールすることも分かった。

なお、ヘルプ表示 (`?`) の副作用として、`?` を入力した時点でコマンドが実行されてしまい、意図せずインストールが開始された。MLNX-OS の `?` は入力中のコマンドを確定せずにヘルプを表示する機能だが、シリアル経由の自動化では `?` 文字の後に改行が送信され、コマンドが確定されたものと思われる。

### 3. `image fetch` の進捗バーとプロンプト検出の衝突

自動化スクリプトは `# ` をプロンプトとして検出していたが、`image fetch` の進捗バーに含まれる `#` 記号と衝突し、ダウンロードが 1.9% の時点で「完了」と誤判定された。実際にはバックグラウンドでダウンロードが継続しており、生のシリアルデータを読み取ることで進行中であることを確認した。完了検知には進捗率の数値監視に切り替えて対処した。

### 4. リブート後のシリアルコンソール無応答

FW 更新後の reload で、シリアルコンソールが全ボーレート (9600, 115200, 38400, 19200, 57600) で無応答になった。mgmt0 への ping は成功していたためスイッチ自体は起動していた。最終的に DTR/RTS シグナルのトグル（`ser.dtr = False` → `ser.dtr = True`）でコンソールが復帰した。PL2303 USB-Serial アダプタの特性と考えられる。

### 5. リブート直後の限定シェル (`CLI > `)

DTR トグル後に得られたプロンプトは通常の `switch-d2b2e2 [standalone: master] >` ではなく `CLI > ` だった。このシェルでは `enable` コマンドも `show version` も使えなかった。ログを確認すると、最初のログインセッションで `cli_hide_cmds` モジュールの初期化に失敗していた（`unexpected NULL` エラー多数）。これはリブート直後のシステム初期化中に前のセッションが残っていたことが原因。`exit` で一度ログアウトし、再ログインすることで正常なフル CLI プロンプトが得られた。

### 6. SSH パスワード認証の一時的失敗

mgmt0 が到達可能になったため SSH 接続も試みたが、`sshpass -p admin` でパスワード認証が失敗した。レガシーな鍵交換アルゴリズム (`diffie-hellman-group14-sha1`) は正しく設定していたが、リブート直後は認証が通らなかった。シリアルコンソールでの操作に切り替えたところ正常にログインできたため、SSH 側のサービス初期化遅延が原因と推測される。

## 手動実施手順

シリアルコンソールに手動接続して FW 更新を行う場合の手順を記載する。自動化スクリプトを使わず、`screen` や `minicom` で直接操作する前提。

### 準備

```sh
# FW イメージをダウンロード
wget -O /tmp/image-PPC_M460EX-3.6.8012.img \
  "https://content.mellanox.com/Software/image-PPC_M460EX-3.6.8012.img"
# MD5: e2114b923351bf4d499c7200392afecb

# server 4 に転送
scp /tmp/image-PPC_M460EX-3.6.8012.img root@10.10.10.204:/tmp/

# server 4 で HTTP サーバを起動 (別ターミナル)
ssh root@10.10.10.204
cd /tmp && python3 -m http.server 8080

# server 4 でシリアルコンソールに接続 (別ターミナル)
ssh root@10.10.10.204
screen /dev/ttyUSB0 9600
```

### Step 1: 設定バックアップ

```
admin / admin でログイン
enable
show running-config          ← テキストを手元にコピー保存
show images                  ← 現在のパーティション構成を確認
show log                     ← mtdblock エラーの有無を確認
```

### Step 2: opensm を停止 (server 4 の別ターミナル)

```sh
systemctl stop opensm
```

### Step 3: FW イメージをスイッチに取得

```
image fetch http://10.10.10.204:8080/image-PPC_M460EX-3.6.8012.img
```

- 進捗バーが表示される。100% まで約 5 分待つ
- 完了するとプロンプトに戻る

### Step 4: FW インストール

```
image install image-PPC_M460EX-3.6.8012.img
```

- 引数なしでデフォルトのパーティション（現在ブート中でない方）にインストール
- 特定パーティションを指定する場合: `image install image-PPC_M460EX-3.6.8012.img location 1`
- 4 ステージの進捗が表示される。約 15 分待つ
- **`partition` キーワードは使えない。`location` を使うこと**

### Step 5: ブート切替・リブート

```
image boot next
show images                  ← Next boot partition が変更されたことを確認
write memory
reload
```

- `Continue to reload the system (y/n)?` に `y` で応答
- リブートに約 8〜10 分かかる

### Step 6: リブート後の確認

- `screen` がフリーズした場合は Ctrl+A, K で終了し、再接続: `screen /dev/ttyUSB0 9600`
- ログインプロンプトが出ない場合は Enter キーを数回押す
- `CLI > ` プロンプトが表示された場合は `exit` → 再ログイン

```
admin / admin でログイン
show version                 ← 3.6.8012 を確認
enable
show images                  ← 更新されたパーティションを確認
show log                     ← mtdblock エラーがないことを確認
```

### Step 7: SM 有効化 (フラッシュ修復成功時)

```
enable
configure terminal
ib sm
ib smnode switch-d2b2e2 enable
exit
```

60 秒待機後:

```
show ib smnode switch-d2b2e2 sm-running    ← "active" を期待
```

- active の場合: `write memory` で永続化
- not active の場合: フラッシュの物理的故障の可能性。SM を無効化して opensm に戻す:
  ```
  configure terminal
  no ib sm
  ib smnode switch-d2b2e2 disable
  exit
  write memory
  ```

### Step 8: 残りのパーティションも更新 (任意)

```
image install image-PPC_M460EX-3.6.8012.img location 1
```

- 両パーティションを同一バージョンにして、フォールバックの信頼性を確保する
- 約 15 分待つ

### Step 9: IB ファブリック復旧 (server 4)

```sh
# スイッチ SM が active の場合、opensm は不要
systemctl disable opensm

# IB 接続確認
ibstat                          # State: Active を確認
ping 192.168.100.2              # IPoIB 疎通確認
```

### Step 10: HTTP サーバ停止 (server 4)

```sh
pkill -f "python3 -m http.server 8080"
```
