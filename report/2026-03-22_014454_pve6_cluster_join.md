# 6号機を pvese-cluster1 (4+5号機) に参加させる

- **実施日時**: 2026年3月22日 01:30 - 01:44 (JST)
- **対象**: 6号機 (ayase-web-service-6, 10.10.10.206)

## 添付ファイル

- [実装プラン](attachment/2026-03-22_014454_pve6_cluster_join/plan.md)

## 前提・目的

- **背景**: 4+5号機が pvese-cluster1 (2ノード quorate)、6号機が pvese-cluster2 (1ノード, Quorum blocked) という状態だった
- **目的**: 6号機を pvese-cluster1 に参加させ、Region A を 3ノード PVE クラスタ (4+5+6号機) にする
- **方針**: OS 再インストールは不要。PVE クラスタ設定をリセットして `pvecm add` で参加。LINSTOR satellite と striped-pool (linstor_vg) は既存のものを再利用

### Phase 1: LINSTOR ノード削除

6号機の LINSTOR satellite をクリーン再登録のため削除。linstor_vg はそのまま保持。

```sh
ssh -F ssh/config pve6 systemctl stop linstor-satellite
ssh -F ssh/config pve4 linstor storage-pool delete ayase-web-service-6 striped-pool
ssh -F ssh/config pve4 linstor node delete ayase-web-service-6
```

### Phase 2: PVE クラスタ設定リセット

pvese-cluster2 の設定を完全除去し、スタンドアロン PVE ノードにする。

**インシデント (前セッション)**: `systemctl stop pve-cluster` 実行後に SSH 接続不能になった。/etc/pve がアンマウントされ、authorized_keys が読めなくなったため。IPMI 経由で power cycle するも、UEFI ブートエントリ消失で起動不能に。→ 別セッションで `efibootmgr -o 0004` でブート順序を修正し復旧。

**今回の対策**: 全操作を1つのスクリプトにまとめて scp → ssh で実行:

```sh
# スクリプト内容 (reset-cluster.sh)
systemctl stop pvedaemon pveproxy pvestatd corosync
systemctl stop pve-cluster
pmxcfs -l &          # ローカルモードで pmxcfs 起動
sleep 2
rm -f /etc/pve/corosync.conf
rm -rf /etc/corosync/*
rm -f /var/lib/corosync/*
rm -f /etc/pve/qemu-server/200.conf
pkill pmxcfs
systemctl start pve-cluster
systemctl start pvedaemon pveproxy pvestatd
```

結果: `pvecm status` → `corosync.conf does not exist` (スタンドアロン化成功)

### Phase 3: pvese-cluster1 に参加

```sh
# SSH 鍵交換 (pve6 → pve4)
# pve6 の公開鍵を pve4 の /root/.ssh/authorized_keys と /etc/pve/priv/authorized_keys に追加
# pve4 の公開鍵を pve6 に追加

# クラスタ参加
ssh -F ssh/config pve6 pvecm add 10.10.10.204 --force --use_ssh
```

結果: `successfully added node 'ayase-web-service-6' to cluster.`

### corosync.conf の `two_node` 削除

3ノードクラスタでは `two_node: 1` は不要。sed で削除し config_version をインクリメント (4→5)。

### Phase 5: LINSTOR satellite 再登録

```sh
ssh -F ssh/config pve6 systemctl start linstor-satellite
ssh -F ssh/config pve6 systemctl enable linstor-satellite
ssh -F ssh/config pve4 linstor node create ayase-web-service-6 10.10.10.206 --node-type Satellite
ssh -F ssh/config pve4 linstor node interface create ayase-web-service-6 ib0 192.168.100.3
ssh -F ssh/config pve4 linstor node set-property ayase-web-service-6 PrefNic ib0
ssh -F ssh/config pve4 linstor node set-property ayase-web-service-6 Aux/site region-a
ssh -F ssh/config pve4 linstor storage-pool create lvm ayase-web-service-6 striped-pool linstor_vg
# LvcreateOptions は scp+ssh パターンで設定 (ダッシュ付き引数の制約)
ssh -F ssh/config pve4 linstor node-connection path create ayase-web-service-6 ayase-web-service-7 cross-region default default
ssh -F ssh/config pve4 linstor node-connection path create ayase-web-service-6 ayase-web-service-8 cross-region default default
ssh -F ssh/config pve4 linstor node-connection path create ayase-web-service-6 ayase-web-service-9 cross-region default default
```

## インシデントと教訓

### 1. pve-cluster 停止で SSH 切断

**原因**: PVE は /etc/pve/priv/authorized_keys を SSH 認証に使用するが、pve-cluster 停止で /etc/pve がアンマウントされる。/root/.ssh/authorized_keys にも鍵があれば回避可能。

**教訓**: クラスタ設定リセットは必ず1つのスクリプトにまとめて scp → ssh で実行する。個別の SSH コマンドでサービスを逐次停止してはならない。

### 2. UEFI ブートエントリ消失

**原因**: 前回セッションでの power cycle 後、UEFI BootOrder に PXE/FlexBoot/EFI Shell が先行していたためディスクブートに到達せず停止。

**対策**: 別セッションで `efibootmgr -o 0004` を実施し、debian のみをブート対象に設定。3台すべてに適用済み。

### 3. DIMM エラー (P2-DIMMA1)

POST 中に `Uncorrectable memory component found: P2-DIMMA1` を検出。現時点では OS は正常起動・動作しているが、今後のモニタリングが必要。
