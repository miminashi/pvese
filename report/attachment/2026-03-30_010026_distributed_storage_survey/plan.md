# 分散ストレージ調査: LINSTOR/DRBD代替の検討

## Context

LINSTOR/DRBDで2リージョン（各3ノード）のマルチリージョンVM運用基盤を構築・評価してきた。
縮退運転（ノード障害時の継続稼働）は可能だが、**1ノード内のディスク冗長性をLINSTOR自体はサポートしていない**ことが判明。
別途RAID/LVM-mirrorで冗長化する必要があり、運用が複雑になる。
ノード内ディスク冗長も組み込みでサポートする分散ストレージを調査した。

## 調査結果サマリ

### 要件を満たす候補: 3つ

| 項目 | Ceph | Vitastor | LINSTOR + LVM-mirror (現行改善) |
|------|------|----------|-------------------------------|
| ノード障害時の縮退運転 | Yes | Yes | Yes |
| ノード内ディスク冗長 | Yes (OSD/disk) | Yes (OSD/disk) | Yes (LVM層で実現) |
| 3ノードでのEC | 非実用的 (host domain) | **可能** (OSD domain) | N/A |
| 4K write latency | ~1ms | ~0.14ms | ~0.15ms |
| CPU使用量 (飽和時) | 40コア/ノード | 4コア/ノード | 中程度 |
| PVE統合 | **組み込み** (pveceph) | 3rdパーティプラグイン | ネイティブプラグイン |
| コミュニティ | 大 (Red Hat支援) | 小 (開発者1名) | 中 |
| マルチリージョン | Stretch cluster (複雑) | 未対応 | **対応済み** |
| ライセンス | LGPL | VNPL 1.1 (copyleft) | GPLv3 |

## 各候補の詳細

### 1. Ceph — 安全な選択肢

**仕組み**: 1物理ディスク = 1 OSD (JBOD, RAID不要)。CRUSHマップの`failure_domain=host`で各レプリカを異なるホストに配置。ディスク障害時はクラスタ全体で自動再配置。

**メリット**:
- PVE組み込みサポート (GUI管理、pveceph CLI、ライブマイグレーション、HA、スナップショット)
- 実績豊富、大規模コミュニティ
- 1ディスク障害 → 自動回復 (同ノードの他OSDに再配置)

**デメリット**:
- 性能: LINSTORの1/2〜1/5 (4K random: read ~8K IOPS, write ~1.5K IOPS)
- リソース消費大 (RAM, CPU)
- 3ノードではEC非実用的 (host domainだとEC 2+1でも3ノード必要で冗長性なし)
- 3ノードレプリケーション(3x): 容量効率33%
- ノード全損後は自動回復不可 (4th hostがない)
- マルチリージョン: Stretch cluster構成が複雑

**3ノード運用時の注意**:
- OSD使用率は45%以下に抑える (回復ヘッドルーム確保)
- `osd_pool_default_size=3`, `min_size=2` で縮退運転

### 2. Vitastor — 高性能な新興候補

**仕組み**: Cephと同じOSD/diskモデルだが、アーキテクチャを大幅に簡素化。etcd + monitorで構成管理。

**メリット**:
- **性能がLINSTOR並み** (4K latency 0.14ms, Cephの7倍速)
- **CPU使用量がCephの1/10** (飽和時4コア vs 40コア)
- 3ノードでもErasure Coding可能 (`failure_domain=osd` で OSD単位の分散)
- EC 2+1: 容量効率67% (Ceph 3xレプリケーションの33%の2倍)
- PVEプラグインあり (ライブマイグレーション、スナップショット対応)
- CRUSH配置の均一性: 96-99% (Cephの80-90%より優秀)

**デメリット**:
- **開発者1名** (Vitaliy Filippov) — バスファクター1
- コミュニティ小 (GitHubスター ~700)
- マルチリージョン未対応
- VNPL 1.1ライセンス (AGPLより厳格だがラボ/プライベート利用は問題なし)
- GUI管理なし (CLI/etcd設定のみ)

### 3. LINSTOR + LVM-mirror — 現行改善

**仕組み**: 現在のLINSTOR構成にLVM-mirrorレイヤを追加。ストレージプールのバックエンドをLVM-mirror (RAID1相当) にする。

**メリット**:
- 既存構成を活かせる (移行コスト最小)
- マルチリージョン対応済み
- 最高性能 (ただしLVM-mirrorのオーバーヘッドあり)

**デメリット**:
- LVM-mirrorの管理が追加される
- ディスク障害時の自動回復はLVM層の手動対応
- 容量効率: ノード内RAID1 + ノード間2xレプリカ = 25%

## 推奨アクション

ユーザの判断ポイントに基づき3つのパスを提示:

### パスA: Ceph (安定重視)
- PVE組み込みで運用が楽、実績豊富
- 性能犠牲あり (LINSTORの1/2〜1/5)
- Region Bの7-9号機で試験導入 → 性能・安定性を評価

### パスB: Vitastor (性能重視)
- LINSTOR並みの性能 + ノード内ディスク冗長
- 3ノードEC対応で容量効率も良い
- リスク: 開発者1名、マルチリージョン未対応
- Region Bで試験導入 → 安定性を長期評価

### パスC: LINSTOR + LVM-mirror (現行改善)
- 移行コスト最小、マルチリージョン維持
- 既に report/2026-03-29_090042_linstor_lvm_raid1_benchmark.md で評価済み

## 検証方法

選択したパスに応じて Region B (7-9号機) に試験導入:
1. インストール・基本設定
2. fio ベンチマーク (4K random read/write, sequential)
3. ディスク障害シミュレーション (OSD停止 → 自動回復確認)
4. ノード障害シミュレーション (1ノード停止 → 縮退運転確認)
5. VM作成・ライブマイグレーション動作確認
